Chapter 3

Digital signatures


For the first time, we will work on a cryptographic notion. Digital signature is a basic protocol, widely used in a lot of softwares dealing with financial transactions or contract management. The concept is quite simple to understand but very efficient.

Digitals signatures tends to prove that a message has been sent by an identifiable sender, it proves also that the message has not been altered (notion of integrity). Plus, it is impossible to deny the fact that the sender sent a message (non repudiation).

The protocol of digitals signatures is composed by three algorithms.
    an algorithm that creates a pair of keys
    an algorithm that signs the message
    an algorithm of verification

We are going to continue our explanation with our ledger for illustration. Why do we need digital signatures?

Imagine that you are sharing your ledger with your roomates, what does protect you from their bad intentions? In fact it could be very easy for them to create false transactions from you to them...

That is why you should sign every transaction from your account in order to validate the fact that you agree with this transaction. In the case of a real ledger you have to trust in the other users because signatures are very easy to falsify. In the case of a cryptocurrency you can trust in the signatures, they will protect you from the other users because it is nearly impossible to imitate them.

So in the case of Coinet, we first use a key creation algorithm. This step will give you a secret key (sk) and a public key (pk) that you will need in the following algorithms. We will not deal with the mathematical sides of the algorithms because it would be very difficult for novices and it would overtake the field of our topic. The important thing to know about this is that keys have to be created in pair because they have to be compatible even if it is impossible to guess one key knowing the other one. The keys have to appear as random numbers.

The public key is transmitted to the ledger and everyone should be able to see it meanwhile the secret key MUST be hidden from the other users.

Logic, isn't it ?

Then after a transaction we should use the signing function. for those of you who are comfortable with basic programming notions, the signing algorithm could be presented like that:

sign(message, sk) = numeric signature.

It means that the algorithm takes in input the message and your secret key (that has to be secret) and outputs a numeric signature. the equivalent of the signature you would put on the ledger.

The signature is a binary number of 256 bits (256 characters 1 or 0). It is possible to verify that the signature is true or if it is a fraud thanks to the function of verification.

Verify(message,signature,pk)= True or False

The verification function takes in input the original message, the signature and the public key in order to validate or not the authenticity of the sender.

the digital signature looks random and it is impossible to find back a sk from the signature. the only way would be to find it by guessing it and verify it. It is nearly impossible because there are too many possibilities with a 256 bit number. To illustrate this, see the bonus chapter about this topic.
